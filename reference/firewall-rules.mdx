description: "Reference for nftables chains, sets, and packet marks generated by luci-app-xray to implement transparent proxying on OpenWrt with firewall4."

luci-app-xray generates its nftables rules at runtime using the ucode template `/usr/share/xray/firewall_include.ut`. The rendered output is written to `/var/etc/xray/01_firewall_include.nft` and injected into the `fw4` `table-pre` table. You do not edit this file directly — it is regenerated whenever Xray restarts or the firewall reloads.

## Policy routing setup

Before any nftables rules take effect, the init script adds policy routing entries so that packets marked `0xfb` (251) are looped back to localhost for TProxy processing:

```bash
ip rule add fwmark 251 lookup 251
ip route add local default dev lo table 251
ip -6 rule add fwmark 251 lookup 251
ip -6 route add local default dev lo table 251
```

Any packet that reaches the TProxy inbound with mark `0xfb` is delivered to the Xray process listening on the configured TProxy ports.

## Traffic marks

These marks are set or checked by the generated nftables rules and by Xray itself.

| Mark (hex) | Mark (dec) | Meaning |
|-----------|-----------|---------|
| `0xfb` | 251 | Packet should be intercepted by TProxy |
| `0xfc` | 252 | Xray direct outbound — dynamic direct connection |
| `0xfd` | 253 | Xray proxy outbound |
| `0xfe` | 254 | Non-IP DNS outbound |
| `0xff` | 255 | User-configured bypass mark (`general.mark`); accepted without further processing |

<Note>
  The bypass mark (`0xff` by default) is configurable via the `mark` option in the `general` UCI section. Xray sets this mark on packets it sends out so that the output hook does not re-intercept them.
</Note>

## nftables sets

The template creates named sets that hold addresses, MAC addresses, and ports used throughout the chains. Sets are created only when the corresponding UCI options are populated.

| Set name | Element type | Purpose |
|----------|-------------|---------|
| `tp_spec_dv4_sp` | `ipv4_addr` (interval) | Private and special-use IPv4 ranges; traffic to these always bypasses TProxy |
| `tp_spec_dv6_sp` | `ipv6_addr` (interval) | Private and special-use IPv6 ranges |
| `tp_spec_sm4_bp` | `ether_addr` | MAC addresses configured for IPv4 bypass in `lan_hosts` |
| `tp_spec_sm6_bp` | `ether_addr` | MAC addresses configured for IPv6 bypass in `lan_hosts` |
| `tp_spec_sm4_tp` | `ether_addr` | MAC addresses configured for IPv4 TProxy in `lan_hosts` |
| `tp_spec_sm6_tp` | `ether_addr` | MAC addresses configured for IPv6 TProxy in `lan_hosts` |
| `tp_spec_dv4_bp` | `ipv4_addr` (interval) | WAN IPv4 destinations to bypass (`wan_bp_ips`) |
| `tp_spec_dv6_bp` | `ipv6_addr` (interval) | WAN IPv6 destinations to bypass |
| `tp_spec_dv4_fw` | `ipv4_addr` (interval) | WAN IPv4 destinations to force-forward (`wan_fw_ips`) |
| `tp_spec_dv6_fw` | `ipv6_addr` (interval) | WAN IPv6 destinations to force-forward |
| `tp_spec_dv4_dg` | `ipv4_addr` (interval) | Default IPv4 gateway addresses (auto-populated by the hotplug script) |
| `tp_spec_dv6_dg` | `ipv6_addr` (interval) | Default IPv6 gateway addresses |
| `tp_spec_dv4_dd` | `ipv4_addr` (timeout) | Dynamic direct IPv4 cache; entries expire after `dynamic_direct_timeout` seconds |
| `tp_spec_dv6_dd` | `ipv6_addr` (timeout) | Dynamic direct IPv6 cache |
| `tp_spec_ctt_dd` | `nfproto . tcp sport` (timeout) | Connection-tracking set for dynamic direct TCP |
| `tp_spec_ctu_dd` | `nfproto . udp sport` (timeout) | Connection-tracking set for dynamic direct UDP |
| `tp_spec_dp4_bp` | `ip daddr . udp dport` | DNS server IP+port pairs that bypass TProxy (avoids intercepting Xray's own DNS queries) |
| `tp_spec_dv4_mt` | `ipv4_addr` | Destination IPs covered by `manual_tproxy` rules |

Per-device forward sets (`tp_spec_sm4_ft_<name>`, `tp_spec_sm4_fu_<name>`, etc.) are created dynamically for each `extra_inbound` referenced from a `lan_hosts` `forward` strategy entry.

## nftables chains

The chains below are injected into the `fw4` `inet` table under the `table-pre` include path. Hook priorities are relative to the standard priority values, offset by `+10` by default (configurable via `firewall_priority`).

### `xray_transparent_proxy`

**Hook:** `prerouting` — type `filter`, priority `filter+10`

This chain is the entry point for packets already marked by a previous traversal. It is intentionally narrow: unmarked TCP and UDP traffic is accepted immediately, keeping the fast path short.

```nft
chain xray_transparent_proxy {
    type filter hook prerouting priority filter+10; policy accept;
    mark 0x000000fb goto tp_spec_wan_fw   # re-entering; route to TProxy
    ip protocol tcp accept
    ip protocol udp accept
    ip6 nexthdr tcp accept
    ip6 nexthdr udp accept
    accept
}
```

### `tp_spec_wan_fw`

Called from `xray_transparent_proxy` when a packet already carries mark `0xfb`. This chain performs the actual `tproxy` action.

Traffic is dispatched to TProxy ports in priority order:

1. **FakeDNS pool** — IPv4/IPv6 TCP and UDP destined for the configured fake-IP pools are redirected to the FakeDNS TProxy ports (defaults: 1086–1089).
2. **Manual TProxy** — destinations covered by `manual_tproxy` source IPs are sent to the standard TProxy ports (1082/1084).
3. **Per-MAC forward** — devices with a `forward` strategy in `lan_hosts` are redirected to the corresponding `extra_inbound` port.
4. **Default** — all remaining IPv4 TCP → port 1082, IPv4 UDP → 1084, IPv6 TCP → 1083, IPv6 UDP → 1085. If no balancer is configured for a direction, the packet is marked with the bypass mark instead.

### `xray_prerouting`

**Hook:** `prerouting` — type `filter` (mangle), priority `mangle+10`

This is the main per-packet decision chain for LAN traffic arriving from `tproxy_ifaces_v4`/`tproxy_ifaces_v6`. It runs before routing and determines whether a packet should be intercepted, bypassed, or passed through.

Processing order:

1. **TTL/hop limit override** — if `ttl_override` or `hop_limit_override` is set, matching packets have their TTL/hop limit rewritten (ports in `ttl_override_bypass_ports` are exempt).
2. **Dynamic direct pass-through** — packets on connections already marked `0xfa` (ct mark) are passed directly, and their source/destination IPs are added to the dynamic direct sets.
3. **Mark `0xfb` pass-through** — packets already marked by `xray_output` are accepted without re-processing.
4. **FakeDNS / manual TProxy check** — jumps to `tp_spec_lan_mf`.
5. **Interface bypass** — interfaces in `bypass_ifaces_v4`/`bypass_ifaces_v6` are accepted immediately.
6. **Per-MAC bypass** — MAC addresses in `tp_spec_sm4_bp`/`tp_spec_sm6_bp` are accepted.
7. **Per-MAC forward** — MAC addresses with a `forward` strategy jump to `tp_spec_lan_ac`.
8. **Per-MAC TProxy** — MAC addresses in `tp_spec_sm4_tp`/`tp_spec_sm6_tp` jump to `tp_spec_lan_ac`.
9. **Interface TProxy** — interfaces in `tproxy_ifaces_v4`/`tproxy_ifaces_v6` jump to `tp_spec_lan_ac`.
10. **Default accept** — all remaining packets are accepted without modification.

### `xray_output`

**Hook:** `output` — type `route` (mangle), priority `mangle+10`

Handles packets originating from the router itself. UID/GID bypass is applied first, then all TCP/UDP traffic jumps to `tp_spec_wan_ac`.

```nft
chain xray_output {
    type route hook output priority mangle+10; policy accept;
    # optional: meta skuid { ... } accept
    # optional: meta skgid { ... } accept
    ip protocol tcp goto tp_spec_wan_ac
    ip protocol udp goto tp_spec_wan_ac
    ip6 nexthdr tcp goto tp_spec_wan_ac
    ip6 nexthdr udp goto tp_spec_wan_ac
    accept
}
```

### `tp_spec_wan_ac`

Called from `xray_output`. Checks Xray's own outbound marks to prevent routing loops:

- Mark `0xfc` → accept (direct outbound; updates dynamic direct IP set)
- Mark `0xfd` → accept (proxy outbound)
- Mark `0xfe` → accept (non-IP DNS)
- Mark `0xff` (or configured `general.mark`) → accept (bypass mark)
- Everything else → jumps to `tp_spec_lan_mf`, then `tp_spec_lan_ac`

If no balancer is configured for a protocol direction, those packets are accepted without interception.

### `tp_spec_lan_mf`

Checks for FakeDNS pool addresses and manual TProxy destinations. Traffic to these addresses jumps directly to `tp_spec_lan_fw` (marking `0xfb`), bypassing the rest of the access control chain. DNS server addresses in `tp_spec_dp4_bp` are accepted here to avoid intercepting Xray's upstream DNS queries.

### `tp_spec_lan_ac`

The access control chain applies IP-level routing decisions:

1. Force-forward IPs (`tp_spec_dv4_fw`/`tp_spec_dv6_fw`) → `tp_spec_lan_fw`
2. Default gateway IPs (`tp_spec_dv4_dg`/`tp_spec_dv6_dg`) → accept
3. Bypass IPs (`tp_spec_dv4_bp`/`tp_spec_dv6_bp`) → accept
4. Private/special IPs (`tp_spec_dv4_sp`/`tp_spec_dv6_sp`) → accept
5. Everything else → `tp_spec_lan_re`

### `tp_spec_lan_re`

Applies port-level policy based on `transparent_default_port_policy`:

- **`forwarded` (default):** bypass port lists (`wan_bp_tcp_ports`, `wan_bp_udp_ports`) are accepted; all remaining TCP/UDP jumps to `tp_spec_lan_dd`.
- **`bypassed`:** forward port lists (`wan_fw_tcp_ports`, `wan_fw_udp_ports`) are sent to `tp_spec_lan_fw`; all other traffic is accepted (bypassed).

### `tp_spec_lan_dd`

Checks the dynamic direct IP cache sets. If the destination IP is cached as dynamic direct, the packet jumps to `tp_spec_lan_ct` for connection tracking. Otherwise the packet is forwarded to `tp_spec_lan_fw` and the connection's source port is added to the connection-tracking set.

### `tp_spec_lan_ct`

Present only when at least one `dynamic_direct_*` option is enabled. If the connection's source port is in the connection-tracking set, it jumps to `tp_spec_lan_fw` (intercepted). Otherwise, `ct mark` is set to `0xfa` — marking the connection as a successfully established dynamic direct connection — and the packet is accepted directly.

### `tp_spec_lan_fw`

The terminal chain for packets that should be intercepted. Sets mark `0xfb` and accepts, which triggers the policy routing rule to deliver the packet to Xray via TProxy.

```nft
chain tp_spec_lan_fw {
    mark set 0x000000fb accept
}
```

## Example: full packet walk

The following describes what happens to a TCP connection from a LAN device to an external IP address under the default configuration (`transparent_default_port_policy forwarded`, one balancer configured).

1. Packet enters `xray_prerouting` from `br-lan`.
2. Not a dynamic direct connection, not already marked — continues.
3. Not a FakeDNS or manual TProxy destination — `tp_spec_lan_mf` returns.
4. Interface matches `tproxy_ifaces_v4` → jumps to `tp_spec_lan_ac`.
5. Not a force-forward, gateway, bypass, or private IP → `tp_spec_lan_re`.
6. Not in bypass port list → `tp_spec_lan_dd`.
7. Not in dynamic direct cache → `tp_spec_lan_fw` → mark set to `0xfb`.
8. Packet re-enters `xray_transparent_proxy` with mark `0xfb` → jumps to `tp_spec_wan_fw`.
9. Not a FakeDNS or manual TProxy destination → default TProxy rule → `tproxy ip to :1082` → accepted.
10. Policy routing table 251 delivers the packet to Xray's `tproxy_tcp_inbound_v4` listener on port 1082.
