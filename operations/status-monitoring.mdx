The `luci-app-xray-status` package extends the LuCI interface with a dedicated status page that gives you a live view of Xray's activity, version details, and connection data. Combined with Xray's built-in metrics, API, and statistics features, you have several options for observing what the proxy is doing at any given time.

## Installing the status package

The status page is provided by a separate package that depends on `luci-app-xray` and `wget`.

```bash
opkg update && opkg install luci-app-xray-status
```

Once installed, navigate to **Services → Xray → Status** in the LuCI web interface. The page displays the package version (built from the git commit hash and a Unix timestamp), along with real-time connection and traffic data when the Xray API is enabled.

## Enabling the Xray API

The status page queries Xray through its gRPC management API. To enable it, set `xray_api=1` in the general UCI section.

When enabled, Xray adds a `dokodemo-door` inbound bound to `127.0.0.1:8080` with the tag `api`. The API exposes three services:

- `HandlerService` — add, remove, or alter inbounds and outbounds at runtime
- `LoggerService` — restart the internal logger
- `StatsService` — query per-user, per-inbound, and per-outbound traffic counters

A routing rule automatically forwards all traffic tagged `api` to the API outbound, keeping it off the regular proxy path.

## Enabling metrics

When you set `metrics_server_enable=1`, Xray opens a Prometheus-compatible metrics endpoint. The default port is `18888`; override it with `metrics_server_port`.

```text
http://<router-ip>:18888/
```

The metrics inbound uses `dokodemo-door` listening on `0.0.0.0` with the tag `metrics`. A corresponding routing rule ensures metrics traffic is handled internally and not forwarded through the proxy.

## Enabling traffic statistics

Set `stats=1` to activate per-user and per-connection traffic accounting. This enables the following policy flags:

| Counter | Scope |
|---|---|
| `statsUserUplink` / `statsUserDownlink` | Per user (policy level 0) |
| `statsInboundUplink` / `statsInboundDownlink` | Per named inbound |
| `statsOutboundUplink` / `statsOutboundDownlink` | Per named outbound |

Statistics are exposed through the `StatsService` API endpoint and are visible on the status page when the Xray API is also enabled.

## Observatory for latency-based balancing

When using the `leastPing` balancer strategy, enable the observatory by setting `observatory=1`. Xray will continuously probe all balancer outbounds and select the one with the lowest round-trip time.

Observatory configuration generated by `gen_config.uc`:

```javascript
{
    subjectSelector: [
        "tcp_balancer_v4@balancer_outbound",
        "udp_balancer_v4@balancer_outbound",
        "tcp_balancer_v6@balancer_outbound",
        "udp_balancer_v6@balancer_outbound",
        "extra_inbound",
        "fake_dns",
        "direct"
    ],
    probeInterval: "100ms",
    probeUrl: "http://www.apple.com/library/test/success.html"
}
```

<Note>
Observatory adds measurable background traffic. Only enable it when you are using the `leastPing` balancer strategy.
</Note>

## Configuring logging

Logging is controlled by three options in the general UCI section:

| Option | Values | Default | Effect |
|---|---|---|---|
| `access_log` | `1` / `0` | `0` | Enable or disable the Xray access log |
| `loglevel` | `debug`, `info`, `warning`, `error` | `warning` | Verbosity of the Xray internal log |
| `dns_log` | `1` / `0` | `0` | Log every DNS query Xray processes |

When `access_log=1`, the `access` field in the generated log config is set to `""` (stdout). When disabled, it is set to `"none"`.

To view Xray log output on the router:

```bash
logread | grep xray
```

For continuous monitoring:

```bash
logread -f | grep xray
```

## Service management

The `xray_core` init script manages the Xray process through procd. It automatically restarts Xray when the `xray_core` or `dhcp` UCI configs change.

```bash
# Start Xray
/etc/init.d/xray_core start

# Stop Xray
/etc/init.d/xray_core stop

# Restart Xray (full stop then start)
/etc/init.d/xray_core restart
```

On start, the init script:

1. Generates `/var/etc/xray/config.json` from UCI settings (up to 5 attempts)
2. Launches the Xray binary under procd with auto-respawn enabled
3. Configures dnsmasq and nftables firewall rules when transparent proxy is enabled

On stop, it removes all firewall and dnsmasq rules that were created at startup.
